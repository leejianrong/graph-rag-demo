The answer often isn't in any single document.

Graph RAG turns a pile of text files into a knowledge graph, then answers multi-hop questions by walking the edges between documents instead of matching keywords inside one of them. This is a small, local demo of that idea. One docker compose up brings the whole stack up on your laptop.

Why vector RAG misses the link

Say you're watching a stream of reports come in. One file mentions that Supplier A is based in Berlin. Another, written months earlier by someone else, notes that a German supply-chain law applies to companies in Berlin. Nobody ever wrote down that the law applies to Supplier A. That fact only exists in the gap between the two documents, two hops apart.

Ordinary vector RAG has a hard time here. It pulls the chunks whose text looks like your question and hands them to a model, and the two paragraphs that matter share no keywords with your question or with each other, so they rarely land in the same retrieval. The link lives in the structure between documents, which is exactly what a bag of chunks throws away and a graph keeps.

Vector RAG

Retrieve by text similarity

Embeds the question, returns the chunks that read most like it. When the bridging sentences don't resemble the question, they never surface, and the model answers from whatever did.

✗ the two-hop link is invisible

Graph RAG

Retrieve by connection

Both files resolve their mention of Berlin to the same graph node. Once that node is shared, a traversal reaches from Supplier A to the law without either file ever naming both.

✓ two hops through a shared node

One document, five stages, in order

Ingestion is a single in-process consumer, not a fan of microservices. A Kafka message carries a bucket and object key, nothing more, and one process runs every stage for that document so you can trace it end to end in a single stack trace. Each stage is still its own swappable module.

01

read

Fetch the document's bytes from object storage using the bucket and key on the trigger message. The raw text is written to Elasticsearch up front, keyed by a deterministic document ID, before any processing runs.

MinIOElasticsearch
02

NER

spaCy extracts typed mentions locally, people, organisations, locations, events, dates, with their character spans, in one pass. It runs on your CPU and costs nothing per token, which matters because it touches every word of every document.

spaCy en_core_web_trflocal · $0
03

coreference

An LLM groups the mentions inside a single document that point at the same thing, pronouns included, and returns a cluster map rather than rewriting the text. The original characters stay put, so the provenance offsets stay valid.

LLM · within-document
04

entity linking

Each cluster is matched against a corpus-local store of canonical entities: blocked by type and normalised name, scored by embedding similarity, then merged into an existing entity or created as a new one. This is the step that lets a mention in one file and a mention in another become one graph node. There's no Wikidata; the corpus is its own authority.

bge-small embeddingslocal-firstES-Entities
05

KG-build

An LLM reads the document and its linked entities and emits subject–predicate–object triples grounded in canonical entity IDs. Predicates map to a closed set of about twelve relations, with a RELATED_TO fallback that keeps the model's original phrasing so nothing is lost. Every edge records the source document and sentence it came from.

LOCATED_INWORKS_FORFOUNDED MEMBER_OFPRODUCESOWNS PART_OFAFFILIATED_WITHRELATED_TO
LLMNeo4j

Dates are the one type that never becomes a node. A graph full of date nodes turns into a hairball, so a date rides along as a qualifier on the edge it belongs to instead.

Retrieval and synthesis are two different jobs

Finding the right entities and evidence for a question can be done with embeddings and a graph traversal alone. Only turning that evidence into prose actually needs a language model, and most Graph RAG demos get quietly expensive by conflating the two.

So the default query path uses no LLM and costs nothing. A synchronous /query call embeds your question, finds seed nodes by vector search over entity and passage embeddings in Elasticsearch, expands a few hops in Neo4j, and returns a ranked subgraph with the supporting sentences and where they came from. For a "who" or "where" question, the answer is the top-ranked node.

embed question→ vector-anchor seeds→ expand k hops (Neo4j)→ ranked subgraph + provenance

An honest limit, not a bug: this path answers entity-typed questions well and explanatory ones poorly. When you want prose, an optional synthesis mode feeds the retrieved subgraph to an LLM. It's gated off by default, so the core path stays free and deterministic.

What comes up under Compose

Everything runs locally, together, from one command. No external knowledge base and no cloud dependency beyond an optional LLM key.

Kafka

The ingestion trigger. Carries {bucket, objectKey} and nothing else.

MinIO

S3-compatible object storage holding the source text and Markdown files.

Elasticsearch

One cluster, two indices: per-document records and canonical entities, which double as the vector index.

Neo4j

The knowledge graph. Multi-label nodes, provenance-carrying edges, k-hop traversal.

spaCy

Local NER over every document. CPU-bound, no API cost.

LiteLLM

Provider-agnostic LLM client. Per-stage model config and a prompt-hash response cache.

bge-small

One local embedding model, reused for entity linking and for query-time anchoring.

FastAPI

Two endpoints: /ingest to upload and trigger, /query to retrieve.

An honest build status

This is a demo built in the open, shipped in vertical slices — from ingestion through the graph to retrieval, optional synthesis, and a benchmark. All eight are in; trust the code over any doc that disagrees.

V1–V5 Landed The full ingestion path. Upload → Kafka trigger → read → NER → coreference → entity linking → KG-build, writing a provenance-carrying graph into Neo4j.
V6 Landed Non-LLM retrieval. The deterministic /query path: vector-anchor, k-hop expansion, ranked subgraph with supporting sentences.
V7 Landed Optional answer synthesis. A gated LLM mode that turns the retrieved subgraph into prose. Off by default.
V8 Landed Benchmarking. A fixed 2WikiMultihopQA subset scored with non-LLM metrics: supporting-fact P/R/F1 and answer exact-match / token-F1.

Run it on your laptop

$ git clone https://github.com/leejianrong/graph-rag-demo
$ cd graph-rag-demo
$ docker compose up        # Kafka + MinIO + Elasticsearch + Neo4j + API
  • Single machine, no cloud. Built to run on a laptop, not to be deployed.
  • The default query path needs no API key. Add an .env LLM key only for the LLM-backed stages and optional synthesis.
  • Any OpenAI-compatible endpoint works, DeepSeek included. Model is a config choice per stage.
  • No authentication, no multi-tenancy, English-only corpus.
